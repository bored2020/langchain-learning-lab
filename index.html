01

WHY LANGCHAIN

它不是模型，
而是模型外面的“应用层”。

一句话定义

LangChain 是一套用于构建 大模型应用 的组件与运行框架： 它负责把模型、提示词、工具、数据和状态组织成一个可执行系统。

A
⌁

只有模型

能生成文字，却不知道你的私有资料，也不能主动查天气、访问数据库或调用业务接口。

封闭的概率引擎
B
⇄

加上 LangChain

统一消息格式、连接外部工具、维护执行状态，并让模型在需要时自主选择下一步。

可组合的应用系统
C
◎

最终得到

问答助手、RAG 知识库、结构化信息提取、自动化工作流，以及能使用工具的 Agent。

面向真实任务
关键区别

Chain 是开发者预先规定步骤；Agent 是模型根据当前上下文决定下一步。刚入门时，两者都要懂，但先从一条可预测的链开始，再进入 Agent。

01

INPUT LAYER

Prompt / Messages

把系统规则、用户问题、历史对话和检索内容，整理成模型能理解的消息列表。它决定模型此刻“看见什么”。

[("system", "你是天气助手"), ("user", question)]
统一接口

为什么能像搭积木？

LangChain 把不同厂商模型、提示模板、检索器和工具包装为相似的可调用对象。 常见对象都支持 invoke()，很多还支持 stream()、batch() 和异步调用。 你因此可以替换组件，而不用重写整套业务逻辑。

三个名字，不要混淆

LangChain高层组件与 Agent 开发入口
LangGraph底层状态图与复杂工作流编排
LangSmith调用链追踪、评测与线上观测

现在先学 LangChain；需要精细控制流程时学 LangGraph；准备调试与上线时接 LangSmith。

03

THE AGENT LOOP

Agent 的本质：
模型在一个循环里调用工具。

执行轨迹 / TRACE
1
USER → AGENT“上海今天穿什么？”

用户提出一个需要实时信息的问题。

2
MODEL → DECISION不能凭空回答，需要查天气

模型根据工具描述，生成结构化工具调用。

3
AGENT → TOOLget_weather(city="上海")

运行时真正执行 Python 函数。

4
TOOL → AGENT“18°C，小雨，东北风 3 级”

工具结果作为新消息放回上下文。

5
MODEL → USER“建议穿薄外套，并带伞。”

模型读到观察结果，组织最终答案并结束循环。

01

INPUT

消息进入状态

Agent 收到的不是一条裸字符串，而是一组带角色的消息。运行时会把新消息加入本次执行状态。

Agent=Model+Harness

Harness 就是模型循环外的一切：提示词、工具、状态与中间件。

!

工具描述就是 Agent 的“说明书”。 函数名、参数类型和 docstring 越准确，模型越容易在正确时机选择正确工具。工具返回的是数据，不应该偷偷夹带新的指令。

04

编写程序

创建 main.py

下面只有四件事：加载配置、定义工具、创建 Agent、发送消息。

MAIN.PY
01import os
02from dotenv import load_dotenv
03from langchain.agents import create_agent
04from langchain.tools import tool
05from langchain_openai import ChatOpenAI
06
07load_dotenv()
08
09@tool
10def get_weather(city: str) -> str:
11    """查询指定城市的天气。"""
12    weather = {
13        "上海": "18°C，小雨，东北风 3 级",
14        "北京": "12°C，晴，西北风 2 级",
15    }
16    return weather.get(city, f"暂时没有{city}的天气数据")
17
18model = ChatOpenAI(
19    model=os.getenv("OPENAI_MODEL", "gpt-5-nano")
20)
21
22agent = create_agent(
23    model=model,
24    tools=[get_weather],
25    system_prompt=(
26        "你是穿衣助手。遇到天气问题必须先调用工具，"
27        "再根据结果给出一句简洁建议。"
28    ),
29)
30
31result = agent.invoke({
32    "messages": [
33        {"role": "user", "content": "上海今天穿什么？"}
34    ]
35})
36
37print(result["messages"][-1].content)

05

DECODE THE CODE

代码很短，
但每一层都在工作。

01@tool 到底做了什么？+

它把普通 Python 函数包装成模型可理解的工具规范，包含名称、用途和参数结构。模型不会直接执行函数，而是先生成“我要调用 get_weather，参数是上海”的结构化请求；Agent 运行时再负责真正执行。

函数签名→JSON Schema→模型决策→Python 执行
02create_agent() 创建了什么？+

它创建的是一个基于 LangGraph 的执行图。核心循环是：调用模型 → 检查是否有工具调用 → 执行工具 → 把结果追加为消息 → 再次调用模型，直到模型产出最终回答。

03invoke() 的输入输出是什么？+

输入是状态字典，这里至少包含 messages。输出仍是状态字典，其中 messages 已追加模型决策、工具结果和最终回答。最后一个消息通常就是交付给用户的 AIMessage。

04为什么示例工具使用假数据？+

为了把变量控制在最少。你此刻只需验证 Agent 是否会选工具、传参数、读取结果。掌握循环后，再把函数体换成真实天气 API；Agent 其余部分不用改变。
