1.1

基本概念

LangChain 解决什么问题

一句话定义

LangChain 是一套用于构建 大模型应用 的组件与运行框架： 它负责把模型、提示词、工具、数据和状态组织成一个可执行系统。

A
⌁

只有模型

能生成文字，却不知道你的私有资料，也不能主动查天气、访问数据库或调用业务接口。

封闭的概率引擎
B
⇄

加上 LangChain

统一消息格式、连接外部工具、维护执行状态，并让模型在需要时自主选择下一步。

可组合的应用系统
C
◎

最终得到

问答助手、RAG 知识库、结构化信息提取、自动化工作流，以及能使用工具的 Agent。

面向真实任务
关键区别

Chain 是开发者预先规定步骤；Agent 是模型根据当前上下文决定下一步。刚入门时，两者都要懂，但先从一条可预测的链开始，再进入 Agent。

01

INPUT LAYER

Prompt / Messages

把系统规则、用户问题、历史对话和检索内容，整理成模型能理解的消息列表。它决定模型此刻“看见什么”。

[("system", "你是天气助手"), ("user", question)]
统一接口

为什么能像搭积木？

LangChain 把不同厂商模型、提示模板、检索器和工具包装为相似的可调用对象。 常见对象都支持 invoke()，很多还支持 stream()、batch() 和异步调用。 你因此可以替换组件，而不用重写整套业务逻辑。

三个名字，不要混淆

LangChain高层组件与 Agent 开发入口
LangGraph底层状态图与复杂工作流编排
LangSmith调用链追踪、评测与线上观测

现在先学 LangChain；需要精细控制流程时学 LangGraph；准备调试与上线时接 LangSmith。

1.3

运行机制

Agent 如何在循环中调用工具

执行轨迹 / TRACE
1
USER → AGENT“广州现在天气怎么样？应该穿什么？”

用户提出一个需要实时信息的问题。

2
MODEL → DECISION生成 tool_calls：city="广州"

模型根据用户消息和工具 Schema 提取城市参数。

3
AGENT → TOOLget_weather(city="广州")

运行时先查广州经纬度，再请求实时天气。

4
TOOL → AGENT{"city":"广州","temperature_celsius":...}

实时结果被包装成 ToolMessage 放回上下文。

5
MODEL → USER根据实时数据给出天气与穿衣建议

模型读到工具结果，组织最终答案并结束循环。

01

INPUT

消息进入状态

Agent 收到的不是一条裸字符串，而是一组带角色的消息。运行时会把新消息加入本次执行状态。

Agent=Model+Harness

Harness 就是模型循环外的一切：提示词、工具、状态与中间件。

1.3.1

参数从哪里来

city="广州" 是谁提取出来的？

get_weather 不会阅读整句用户问题。城市识别发生在模型生成工具调用时，LangChain 随后负责校验参数并执行函数。

模型 / Model 理解并提取

从“广州现在天气如何”中识别出城市是“广州”。

运行时 / LangChain 校验并调度

读取模型生成的工具名与参数，找到对应工具并调用。

函数 / Tool 接收并查询

收到已经整理好的 city，再访问真实天气服务。

模型看到的工具说明

下面这份 Schema 来自函数名、类型注解和 docstring。它告诉模型：调用工具时必须提供一个字符串城市名。

TOOL SCHEMA
{
  "name": "get_weather",
  "description": "查询指定城市的实时天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {"type": "string"}
    },
    "required": ["city"]
  }
}

动手观察

先只让模型生成调用，不执行工具

bind_tools() 会把工具说明交给模型。打印 tool_calls，你就能看到“广州”是在函数执行前由模型生成的。

预期输出 [{'name': 'get_weather', 'args': {'city': '广州'}}]
INSPECT_TOOL_CALL.PY
model_with_tools = model.bind_tools([get_weather])

message = model_with_tools.invoke(
    "广州现在天气怎么样？"
)

print(message.tool_calls)
缺少城市时怎么办？

如果用户只说“今天天气如何”，模型不应该猜城市。系统提示应要求它先追问：“你想查询哪个城市？” 工具参数只能来自用户消息、对话上下文或应用明确提供的位置数据。

!

工具描述就是 Agent 的“说明书”。 函数名、参数类型和 docstring 越准确，模型越容易在正确时机选择正确工具。工具返回的是数据，不应该偷偷夹带新的指令。

04

编写程序

创建 main.py

代码中的 ①–⑦ 依次展示：工具 Schema 如何生成、广州如何进入参数、真实 API 如何被调用，以及 ToolMessage 如何回到模型。

MAIN.PY
import json
import os

import requests
from dotenv import load_dotenv
from langchain.agents import create_agent
from langchain.tools import tool
from langchain_openai import ChatOpenAI

load_dotenv()

GEOCODING_URL = "https://geocoding-api.open-meteo.com/v1/search"
FORECAST_URL = "https://api.open-meteo.com/v1/forecast"
WEATHER_CODES = {
    0: "晴朗", 1: "大致晴朗", 2: "局部多云", 3: "阴天",
    45: "有雾", 61: "小雨", 63: "中雨", 65: "大雨",
    80: "小阵雨", 81: "中等阵雨", 82: "强阵雨", 95: "雷雨",
}

# ① @tool 读取函数名、docstring 和 city: str，生成工具 Schema。
@tool
def get_weather(city: str) -> str:
    """查询指定城市的实时天气。

    Args:
        city: 用户明确提出的城市名，例如“广州”。
    """
    city = city.strip()
    if not city:
        return json.dumps({"error": "城市名不能为空"}, ensure_ascii=False)

    try:
        # ② 函数收到的已是 city="广州"，先将城市转换成经纬度。
        location_response = requests.get(
            GEOCODING_URL,
            params={"name": city, "count": 1, "language": "zh", "format": "json"},
            timeout=10,
        )
        location_response.raise_for_status()
        locations = location_response.json().get("results", [])

        if not locations:
            return json.dumps({"error": f"没有找到城市：{city}"}, ensure_ascii=False)

        location = locations[0]

        # ③ 使用广州的经纬度请求实时天气，没有写死温度和天气。
        weather_response = requests.get(
            FORECAST_URL,
            params={
                "latitude": location["latitude"],
                "longitude": location["longitude"],
                "current": (
                    "temperature_2m,apparent_temperature,"
                    "precipitation,weather_code,wind_speed_10m"
                ),
                "timezone": location["timezone"],
            },
            timeout=10,
        )
        weather_response.raise_for_status()
        current = weather_response.json()["current"]
    except requests.RequestException as error:
        return json.dumps(
            {"error": f"天气服务请求失败：{error}"},
            ensure_ascii=False,
        )

    # ④ 只返回结构化事实；Agent 会把字符串包装成 ToolMessage。
    return json.dumps(
        {
            "city": location["name"],
            "province": location.get("admin1"),
            "observation_time": current["time"],
            "weather": WEATHER_CODES.get(
                current["weather_code"],
                f"天气代码 {current['weather_code']}",
            ),
            "temperature_celsius": current["temperature_2m"],
            "apparent_temperature_celsius": current["apparent_temperature"],
            "precipitation_mm": current["precipitation"],
            "wind_speed_kmh": current["wind_speed_10m"],
        },
        ensure_ascii=False,
    )

model = ChatOpenAI(
    model=os.getenv("OPENAI_MODEL", "gpt-5-nano")
)

# ⑤ 注册工具：Schema 给模型看，Python 函数交给 Agent 运行时。
# create_agent 内部的派发过程可理解为下面这段伪代码：
# tools_by_name = {get_weather.name: get_weather}
# call = ai_message.tool_calls[0]                 # args={"city": "广州"}
# selected_tool = tools_by_name[call["name"]]    # 按名称找到 Python 函数
# tool_result = selected_tool.invoke(call["args"])  # 传入 city="广州"
# messages.append(
#     ToolMessage(content=tool_result, tool_call_id=call["id"])
# )  # 把结果与本次调用关联，再交给模型
    model=model,

agent = create_agent(
    tools=[get_weather],
    system_prompt=(
        "你是天气与穿衣助手。天气问题必须调用 get_weather，"
        "不能根据记忆猜测。没有城市时先追问。"
    ),
)

# ⑥ 模型读到“广州”，生成 get_weather(city="广州")。
result = agent.invoke({
    "messages": [{
        "role": "user",
        "content": "广州现在天气怎么样？应该穿什么？",
    }]
})

# ⑦ 打印 HumanMessage → tool_calls → ToolMessage → 最终 AIMessage。
for message in result["messages"]:
    tool_calls = getattr(message, "tool_calls", None)
    if tool_calls:
        print("模型生成工具调用：", tool_calls)
    elif type(message).__name__ == "ToolMessage":
        print("工具返回实时数据：", message.content)

print("最终回答：", result["messages"][-1].content)

1.5

代码解析

理解每一层正在做什么

01@tool 到底做了什么？+

它把普通 Python 函数包装成模型可理解的工具规范，包含名称、用途和参数结构。模型不会直接执行函数，而是根据“广州现在天气怎么样”生成 {"name":"get_weather","args":{"city":"广州"}}；Agent 运行时再负责真正执行。

函数签名→JSON Schema→模型决策→Python 执行
02create_agent() 创建了什么？+

它创建一个基于 LangGraph 的执行图，并建立工具名称到 Python 函数的映射。核心循环是：调用模型 → 检查 AIMessage.tool_calls → 用 args={"city":"广州"} 调用工具 → 追加 ToolMessage → 再次调用模型，直到没有新的工具调用。

03invoke() 的输入输出是什么？+

输入是包含 HumanMessage 的状态字典。输出仍是状态字典，但 messages 已依次追加带 tool_calls 的 AIMessage、包含实时 JSON 的 ToolMessage，以及最终 AIMessage。代码中的循环把这四类消息逐条打印出来。

04get_weather("广州") 内部做了什么？+

函数先请求地理编码服务，把“广州”转换为纬度、经度和时区；再用这些坐标请求实时天气。它最终返回 JSON 字符串，而不是自己编写穿衣建议。事实由工具提供，表达与建议由模型完成。

广州→23.11667, 113.25→实时天气 JSON→ToolMessage
05为什么工具返回 JSON，而不是一段漂亮文字？+

JSON 能保留城市、观测时间、温度、体感温度、降水和风速的字段边界，方便模型和程序可靠读取。工具负责返回可验证的事实，模型再根据系统提示把事实组织成人类易读的回答。
